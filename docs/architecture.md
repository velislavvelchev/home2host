# Architecture

## What we're rebuilding

`home2host.com` currently runs on WordPress + Elementor + TranslatePress. The site is essentially a marketing site + blog; the "Our apartments" section is not a real property database вЂ” it's a collection of embedded Airbnb listings. So the migration is mostly: rebuilding static pages as React components, moving the blog into a CMS, and re-embedding the Airbnb listings.

## Stack

| Layer | Choice | Why (short version) |
|---|---|---|
| Frontend | Next.js (App Router) + TypeScript | React-based, SSR/SSG for speed and SEO, trivial deployment |
| CMS | Payload, same repo as the frontend | Admin panel for the business partner, schema-as-code, single stack |
| Database | Postgres on Neon ([ADR 0002](decisions/0002-database-provider.md)) | Required by Payload; serverless, Vercel-native, scale-to-zero |
| Hosting (frontend) | Vercel (free tier) | Built by the Next.js team, GitHub integration in minutes |
| Domain / DNS | Hostinger remains the registrar | Only DNS is repointed to Vercel at the very end |
| i18n | next-intl | Replaces TranslatePress; UI strings in JSON, blog uses bilingual fields in CMS |
| Analytics | Google Analytics 4 via `@next/third-parties` | Same measurement ID вЂ” historical data in GA is preserved |
| Version control | Git + GitHub | One repo for everything |

## What we replace

- Elementor в†’ React components
- TranslatePress в†’ next-intl
- WordPress wp-admin в†’ Payload admin panel

## What we explicitly do NOT replace (for now)

- The Airbnb listings stay as embed widgets вЂ” no custom "apartment" entity with filters/search. If a real database of properties becomes necessary later, it's a separate feature, not part of this migration.

## Migration safety

The WordPress site stays live on the domain throughout development. The new site is built on a temporary Vercel URL (something like `home2host.vercel.app`). DNS is repointed **only at the very end**, after full verification. WordPress is **not deleted** after launch вЂ” it's kept as a rollback option for several weeks.

## Project layout

Payload and Next.js share the `src/app/` tree via **parallel root layouts** вЂ” a Next.js App Router pattern that lets two route groups each own their own `<html>/<body>`. This is non-negotiable: collapsing them into a single `src/app/layout.tsx` produces nested `<html>` tags and hydration errors (we learned this the hard way during Stage 1).

**Where new things go:**

- Frontend pages (About, Services, Blog, etc.) в†’ `src/app/(frontend)/<route>/page.tsx`
- Shared components в†’ `src/components/` (created on first real component, not pre-emptively, per [ADR 0003](decisions/0003-styling-approach.md))
- Payload collections / globals в†’ added inside `src/payload.config.ts`
- After any change to `payload.config.ts`:
  - Run `npm run generate:types` so `src/payload-types.ts` stays in sync.
  - If the change adds or removes a field that uses a custom component (richText/Lexical, custom field UI, etc.), also run `npm run generate:importmap` to refresh `src/app/(payload)/admin/importMap.js`. Skipping this surfaces as `PayloadComponent not found in importMap` errors in the admin (learned the hard way during Stage 2).

## Local development

Fresh clone:

```powershell
npm install
vercel link                          # links the directory to the Vercel project
vercel env pull .env.local            # pulls DATABASE_URL + PAYLOAD_SECRET for dev
npm run dev                           # localhost:3000 (admin at /admin)
```

`PAYLOAD_SECRET` is generated locally on first setup, then synced to the project via `vercel env add` so prod and dev share the same signing key. The free Neon tier has **no per-branch database**, so local dev hits the same DB as production вЂ” be deliberate about which content you create where.

**Payload CLI quirk on Node 22+:** the `payload` bin uses `tsx` to load `payload.config.ts`, which crashes with `ERR_REQUIRE_ASYNC_MODULE` when the config graph contains top-level await (it does). The `--use-swc` flag in our `generate:*` scripts swaps `tsx` for `@swc-node/register`, which handles ESM with TLA correctly. **Do not remove `--use-swc` from those scripts** without testing first on your Node version.
