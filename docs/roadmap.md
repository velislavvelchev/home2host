## Stage 4 — Pages 🔄

**Architecture:** the new site is a multi-page app with a rich home page that carries the **full** content of every section (preserves the live home's existing Google ranking authority). Each section also lives at its own URL — slugs matching the live WordPress site exactly (`/about-us/`, `/services/`, `/prices/`, `/questions/`, `/apartments/`, `/contacts/`, `/blog/`, trailing slashes preserved via `trailingSlash: true` in `next.config.ts`). Standalone section pages set `<link rel="canonical" href="/">` so they don't compete with the home for the same keywords. This is essentially what the live WordPress site already does (most standalone pages repeat home-page content), but made explicit.

Section components live under `src/components/sections/` and are reused 1:1 between the home embed and the standalone route — heading level swaps via a `headingLevel` prop so the document outline stays correct in both contexts.

Order: static pages first, then CMS-driven ones.
- ✅ Home, About, Services, Prices, FAQ, Contacts — all six sections built, embedded on `/`, and accessible as standalone routes with canonical → `/`.
  - ✅ Header/Footer nav aligned with live slugs + BG labels; `trailingSlash: true` set so URLs match the WordPress shape exactly.
  - ✅ About section (`src/components/sections/AboutSection.tsx`) — first real section, BG content from inventory, embedded on `/` and standalone at `/about-us/` (canonical to `/`).
  - ✅ Hero rebuild — placeholder replaced with BG title/eyebrow/tagline + mid-century interior photo (`public/hero-home.jpeg`) served via `next/image` with `priority` (preload link in head, srcSet from 384–3840px). Ambient liveliness via two CSS-only gradient blobs (offset 4s for out-of-phase pulsing) + slow Ken Burns zoom on the image (1.00 → 1.06 over 24s). All animations are `motion-safe:` gated so users with `prefers-reduced-motion` get the static version. Polish (more pronounced liveliness) parked as a follow-up.
  - ✅ Brand identity wired through — `public/logo.svg` (full lockup, vector from owner) + `public/logo-icon.svg` (cropped icon-only variant for the Header/Footer chip + SVG favicon) + 3 PNG favicon sizes + OG share image. Page metadata uses BG title/description, `lang="bg"`, `locale: "bg_BG"`. Geist Sans/Mono subsets now include `cyrillic` so BG copy renders in Geist instead of falling back to a system font.
  - ✅ Services section (`src/components/sections/ServicesSection.tsx`) — editorial alternating image/text rows (image left/right on `md+`, stacked on mobile), Lucide icon + `NN / 06` indicator per row, six Pexels photos under `public/services/` (placeholder until owner photography). Embedded on `/` and standalone at `/services/` (canonical to `/`).
  - ✅ Prices section (`src/components/sections/PricesSection.tsx`) — three pricing cards (Start Smart / Full Care / Home Refresh) with distinct Lucide icons, split-typography numeric prices, brand-tinted Check icons, cascading RevealOnScroll.   - ✅ FAQ section (`src/components/sections/FaqSection.tsx`) — 7 Q&A pairs as a native `<details>/<summary>` accordion with chevron rotation, hairline dividers, single-column reading layout. Embedded on `/` and standalone at `/questions/` (canonical to `/`). "BnB Manager" typo from live site corrected to "Home2Host".
  - ✅ Contacts section (`src/components/sections/contacts/`) — two-column layout with service area + phone/email/address/WhatsApp/social on the left and a 4-field form on the right. Form posts via Next.js Server Action through Hostinger SMTP to the existing `info@home2host.com` mailbox; `Reply-To` set to the submitter so owner replies in webmail land directly. Honeypot field included; **per-IP rate limit shipped 2026-06-16** (`@upstash/ratelimit` sliding window, 5/hour, fails open if env vars unset — see [contact-form abuse defenses](#stage-4--pages-) below for the full note). Header transitions cleanly across the `lg:hidden` hamburger threshold; Apartments carousel arrows sit nicely in the gutter at 768+; contact form 2-col split at md+ holds. Real-device pass on actual Android/iPhone still deferred to Stage 6 — emulator/DevTools doesn't cover Safari-specific viewport-height + tap-target quirks.
- ✅ Contact-form abuse defenses — **honeypot shipped** with field name `h2h_confirm` plus password-manager opt-out attributes (renamed from `website` 2026-06-16 after Chrome's autofill was found to trip the trap on legit users). **Rate limiter live in production** (2026-06-16): `@upstash/ratelimit` + `@upstash/redis` sliding window, 5 submissions per IP per hour, runs after the honeypot and before validation/SMTP so invalid payloads still throttle. IP read from `x-forwarded-for` (first hop) with `x-real-ip` fallback. Fails open with a console warning if env vars are missing, so local dev works without an Upstash account. Upstash Redis provisioned on EU-Central, `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` set on Vercel Production/Preview/Development, end-to-end verified against `home2host.vercel.app`. Optional Cloudflare Turnstile left as residual-spam follow-up if needed post-launch.

  - **Slice 1** (commit `be706bd`): plumbing — `next-intl` 4.13 + middleware + `[locale]` route + `setRequestLocale` + `NextIntlClientProvider`. Build prerenders 34 pages × 2 locales; `/admin` and `/sitemap.xml` untouched.
  - **Slice 2** (commit `461175b`): extract every hardcoded BG string into `messages/bg.json` (per-component namespaces, arrays via `t.raw()`, Hero highlight via `t.rich`). Per-page metadata switched to async `generateMetadata` calling `getTranslations({locale, namespace})`. Contact-form server action: submitter-facing messages route through next-intl in the visitor's locale; email body to the owner stays hardcoded BG. Side-fix: apartment cards' city labels now resolve through `Apartments.cities` instead of rendering the raw English string on BG pages.
  - **Slice 3** (this commit): EN content drafted into `messages/en.json` (MT-quality, owner refines in place). `LanguageSwitcher` actually swaps locales via `next-intl/navigation`'s `useRouter` + `usePathname` with `useTransition` for responsiveness. `src/i18n/navigation.ts` exports locale-aware `Link`/`useRouter` wrappers; swapped `next/link` → `@/i18n/navigation` Link across every internal navigation site so EN visitors stay in EN as they click around. Sitemap doubled with hreflang `alternates.languages` per static URL pair (22 URL entries, 38 hreflang lines verified). Blog locale policy: posts probe Payload with `fallbackLocale: false` to detect translation status; untranslated posts on the EN routes carry an "Only in Bulgarian" badge on the card and an inline notice on the detail, both auto-clearing when the owner saves an EN title.
- ✅ GA4 with the same measurement ID as the old site — `@next/third-parties` installed and wired into the frontend layout. `NEXT_PUBLIC_GA_MEASUREMENT_ID` set on Vercel Production + Preview to `G-5HEDRYZJ93` (the same measurement ID the live WordPress site uses via Site Kit), so analytics continuity is preserved through the Stage 6 DNS switch. Verified in deployed page source 2026-06-11.
- ✅ Sitemap (`/sitemap.xml`) — dynamic, lists every public marketing page; appends each published blog post's URL with `lastModified` from `updatedAt`. Payload-read failures are swallowed so a transient DB issue can't blank the static portion.
- ✅ `robots.txt` — allow public content, disallow `/admin/` + `/api/`, point at the sitemap.
- ✅ JSON-LD `LocalBusiness` structured data — rendered in the layout body.
- ✅ Per-page Open Graph metadata — each section route (`/about-us/`, `/services/`, `/apartments/`, `/prices/`, `/questions/`, `/contacts/`) and `/blog/` overrides the root layout's `openGraph.title` + `description` with its own copy. `images` intentionally inherits the root's OG image — making per-section share images is a separate design slice. Per-post OG on `/blog/[slug]/` set via `generateMetadata` (title, description from excerpt, `og:type=article`, image from `featuredImage`).
- ✅ Redirects from old URLs (preserve Google rankings) — legacy WordPress system URLs covered (`/wp-admin/*`, `/wp-login.php`, `/author/*`, `/category/*`, `/tag/*`, `/feed`, `/feed/*`, `/comments/feed` → all 308 to `/`). Marketing-page slugs already match the live WP shape thanks to `trailingSlash: true`. Five blog posts imported into Payload under their original WordPress slugs (Strategy A); the sixth (the WP `-copy` draft duplicate) was given a clean English slug `eu-regulation-2024-1028-what-to-know` with a 308 from the legacy Cyrillic URL → new URL.
- ⬜ **Editor-facing SEO assessment in Payload admin** (Yoast-style) — real-time scoring panel inside the admin so the partner sees how a blog post (or any text-heavy entity) stacks up on SEO + readability **as they edit**, not after publishing. Target signals: focus keyphrase presence in title/H1/slug/meta/first paragraph, keyword density, meta-description length, image alt-text coverage, internal-link presence, readability (sentence length, paragraph length, transition-word frequency). **Needs a build-vs-adopt research pass first**: check if a maintained Payload v3 plugin exists; if not, build a custom admin sidebar component (Payload supports this via `admin.components` on collection configs). BG-locale readability is the hard part — most existing analysers are English-only, so we may need to either roll a simple BG-aware analyser ourselves or ship English-only scoring as V1 and add BG later. Applies to `blog-posts` first; extend to `apartments` (after the Stage 1 follow-up sync), `services`, `faq`, `pricing` as a follow-up. **Owner-requested 2026-06-16**: in-scope before launch since content quality at launch is what Google indexes first.

## Known follow-ups (don't lose these)

Small items deferred during Stage 1 — none block Stage 2, but each will bite at a specific later moment if forgotten.

- ✅ **Payload `Apartment` collection out of sync with the new card design** (closed 2026-06-18). Sync landed as part of Priority B below: dropped the embed-era `description` field, added `featuredImage` (Media upload, required) and `rating` (nullable number 0–5), updated the admin copy. Frontend reads live from the collection now.

- **`PAYLOAD_SECRET` missing from Vercel Preview env.** Production and Development have it; Preview does not (Vercel CLI 54.9.1 wouldn't accept it non-interactively, and we had no PR workflow yet). Add it via Vercel dashboard → Settings → Environment Variables before the first PR-triggered preview deploy, otherwise `/admin` on the preview URL will 500 with "missing secret key".
- **`pg-connection-string` v3.0 / pg v9 SSL semantics.** The dev log shows a warning that `sslmode=require` currently aliases to `verify-full`, but won't in pg v9. When we upgrade to pg v9, switch the Neon URL to `sslmode=verify-full` explicitly. Until then: no action.
- **Email adapter not wired.** Payload currently writes emails to console. Add Resend or SMTP adapter in Stage 4 or 5, before any production user needs a password reset. Until that exists, **don't lose the admin credentials** — no self-serve recovery.
- **Neon DB branching not enabled.** Per [ADR 0002](decisions/0002-database-provider.md), enable preview-deploy branching in Stage 5 or 6 once there's real production content worth protecting. Until then dev and prod share the same database.

- **Payload admin UI not brand-themed.** Currently ships with Payload's stock indigo/grey palette. The partner will spend real time inside the admin during content entry; matching it to the site's `#122C69` brand-800 indigo + Geist typography makes the system feel like one product instead of two. Entry point already exists: `src/app/(payload)/custom.scss` is wired into Payload's admin config. Scope: override the relevant `--theme-*` CSS variables (elevation tokens for buttons + chrome, brand accent for success/active states), align the admin font with Geist Sans, optionally drop the Home2Host logomark into the admin header. Half-day slice; owner-requested 2026-06-16, ship pre-launch.

- **Marketing sections sourced from JSON, not from their Payload collections.** Surfaced 2026-06-17 during the Stage 5 i18n migration: Payload has collections defined for FAQ, Service, PricingPlan, Apartment (created in Stage 2), but the frontend reads from `messages/<locale>.json` for every marketing section. Net effect: collections exist in the admin but sit empty, and the owner can't self-serve content updates that look "editable" — they need a developer to ship a JSON commit. Two-tier triage:
  - ✅ **Priority A — wire FAQ collection → frontend** (2026-06-18). `FaqSection` is now an async server component that reads from `payload.find({ collection: 'faqs', locale, sort: 'order' })`. The 7 existing Q&As live in the DB (seeded once via a temporary `/api/seed-faqs` Next.js endpoint — a standalone `tsx` script tripped Payload's bundled `loadEnv.js` interop, and the in-app endpoint sidesteps the entire loader fight). Owner now self-serves edits in `/admin` exactly like blog posts. `Faq.items` removed from `messages/<locale>.json`; only eyebrow/heading/lead/meta remain as section chrome. Trade-off: the home + `/questions/` routes flipped from prerendered to dynamic so they can read the DB per request — fine for current traffic, revisit with `unstable_cache` if needed.
  - ✅ **Priority B — wire Apartment collection** (2026-06-18). `ApartmentsSection` is now an async server component reading from `payload.find({ collection: 'apartments', where: { isActive: { equals: true } }, sort: 'order', depth: 1 })`. The 10 hardcoded listings were migrated into the DB via a one-shot `/api/seed-apartments` endpoint that downloaded each og:image and uploaded it to Vercel Blob as a Media doc + featuredImage relation. Schema sync landed in the same slice (see closed follow-up above). **Beyond the "wire to Payload" baseline, also shipped a "Fetch from Airbnb" admin button** (`src/components/admin/FetchFromAirbnbField.tsx` + `/api/fetch-airbnb`): owner pastes a public Airbnb URL → clicks Fetch → server-side parses `og:image` + JSON-LD `name` + `aggregateRating.ratingValue`, downloads the cover photo, uploads to Blob, returns title/mediaId/rating which the React component writes into the form via `useField`. Plus a smaller refresh-arrow icon button (`RefreshAirbnbRatingField` + `/api/refresh-airbnb-rating`) that pulls just the rating — useful for the most-volatile field without overwriting a manually-curated title or photo. Both endpoints are auth-gated via `payload.auth({ headers })`; accept any Airbnb subdomain (`www.airbnb.com`, `bg.airbnb.com`, `www.airbnb.co.uk`, etc.); degrade gracefully when Airbnb changes their HTML (graceful error in the admin, owner falls back to manual entry). The `scripts/fetch-airbnb-og-images.mjs` one-shot script is deleted — superseded by the in-admin flow. `Apartments.ratingNew` i18n key dropped (cards now hide the pill entirely when rating is null rather than showing a stale "New" snapshot).
  - **Leave in JSON for now: Services, PricingPlans, Hero, About, Footer, Contacts copy.** These rarely change; "email the developer for an update" is reasonable cadence; the wiring cost outweighs the runtime benefit. Revisit only if the owner asks for self-serve editing of any of them.
